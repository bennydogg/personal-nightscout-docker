## Project Overview

Personal Nightscout CGM (Continuous Glucose Monitoring) deployment using Docker. Runs the Nightscout web application (v15.0.6) with MongoDB 4.4, exposed via Cloudflare Tunnel for secure external access. Supports multiple instances on a single server via per-user directories.

## Architecture

Two Docker containers per instance, orchestrated via docker-compose:
- **nightscout**: `nightscout/cgm-remote-monitor:15.0.6` - CGM data visualization web app (internal port 1337, mapped to configurable `HOST_PORT`)
- **mongo**: `mongo:4.4` - Data persistence with named volume, WiredTiger cache capped at 0.25GB by default

Both containers share a bridge network. Port binding is localhost-only (`127.0.0.1:HOST_PORT:1337`). External access is via Cloudflare Tunnel (systemd service, not Docker container). Container names are auto-generated by compose project name (no hardcoded `container_name`).

### Multi-instance layout

Each user gets their own directory with independent compose project, `.env`, and data volume:
```
/opt/nightscout/
  alice/    # docker-compose.yml, .env (HOST_PORT=8081)
  bob/      # docker-compose.yml, .env (HOST_PORT=8082)
```
One Cloudflare tunnel with multiple ingress rules routes subdomains to different local ports.

## Common Commands

```bash
# Create/migrate an instance
./migrate-instance.sh --name alice --domain alice-ns.example.com --port 8081
./migrate-instance.sh --name bob --domain bob-ns.example.com --port 8082 --source /old/instance

# Per-instance management (run from instance directory)
docker-compose up -d
docker-compose down
docker-compose logs -f
./validate.sh

# Initial single-instance setup
./setup.sh --domain nightscout.yourdomain.com --setup-tunnel
```

## Configuration

All configuration via `.env` file per instance. Key variables:
- `HOST_PORT` - Unique port per instance (8080, 8081, etc.)
- `API_SECRET` - Nightscout API authentication (min 12 chars)
- `MONGO_INITDB_ROOT_PASSWORD` - MongoDB root password
- `MONGO_CONNECTION` - Connection string (database name: `nightscout`)
- `MONGO_CACHE_SIZE_GB` - WiredTiger cache limit (default 0.25, important for multi-instance RAM)
- `CLOUDFLARE_DOMAIN` / `CLOUDFLARE_TUNNEL_ID` - Tunnel config

## Script Categories

**Setup**: `setup.sh` (main), `setup-cloudflare.sh` (tunnel), `migrate-instance.sh` (multi-instance)
**Validation**: `validate.sh`, `validate-database.sh`, `validate-migration.sh`
**Tunnel debugging**: `debug-tunnel.sh`, `fix-tunnel.sh`, `cleanup-tunnels.sh`
**Migration**: `export-atlas-db.sh` (from Atlas), `import-to-vm.sh` (to local), `migrate-instance.sh` (instance-to-instance)
**Maintenance**: `cleanup.sh` (scoped to current compose project), `backup.sh`, `diagnose.sh`

All scripts are bash, use `set -e`, and share a common colored output pattern (`print_status`, `print_warning`, `print_error`, `print_info`).

## Known Technical Debt

- MongoDB 4.4 is EOL (Feb 2024). Upgrade to 7.0 requires stepping through 5.0 and 6.0 with `featureCompatibilityVersion`. Plan separately from other changes.
- `backup.sh` has issues: `mongodump` missing auth flags, `read -p` prompts hang in cron, references non-existent `restore.sh`. Needs rewrite before relying on it.
- `mongo:4.4` uses legacy `mongo` shell, not `mongosh`. Scripts must use `mongo` for exec commands.
